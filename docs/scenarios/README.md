# GoTo Connect Demo Training — Scenario Guide
**For:** Solution Consultant Training
**Customer Role:** Taylor, owner of Shear Bliss Salon
**SC Role:** Solution Consultant demoing GoTo Connect

---

## Overview

These 5 roleplay scenarios train Solution Consultants to run live GoTo Connect demos. Each scenario starts from a real business pain point and builds naturally toward a product demo. Scenarios progress from basic to advanced — they can be run individually or as a full arc in order.

The goal is not just to teach features. It's to train SCs to **listen first, demo second** — and to always connect what GoTo Connect does to what the customer's business actually needs.

---

## Customer Profile: Taylor

Taylor is the owner of **Shear Bliss Salon** — a small, 6-person salon with a mix of employees and booth renters. He's friendly, direct, and practical. He's not technical and doesn't know VoIP terminology. He cares about his staff, his clients, and growing his business.

| | |
|---|---|
| **Name** | Taylor |
| **Business** | Shear Bliss Salon |
| **Staff** | 6 (grows to 8 by Scenario 5) |
| **Personality** | Warm, business-savvy, non-technical, motivated by growth and staff wellbeing |
| **Pain sensitivity** | Responds to staff experience and client retention arguments |
| **Decision style** | Makes decisions when he can see the problem clearly and the solution feels manageable |

**Taylor's language:** He says "clients," not "customers." She talks about "bookings," not "leads." He says "my team" or "my stylists," not "employees." Match him language.

## How to Use These Scenarios

### Running a Single Scenario
Each scenario file is self-contained. Hand the SC the scenario file, give them a few minutes to read the Business Context section, then start the roleplay. The trainer plays Taylor using the pain points listed — don't read from the sample dialogue, use it as a guide for how the conversation should feel.

### Running All 5 as a Full Arc
The scenarios are designed to build on each other. Taylor's situation evolves:
- Scenario 1: No phone system at all
- Scenario 2: Has a number but no visibility
- Scenario 3: Has basic setup but texting is a problem
- Scenario 4: Wants a professional experience (competitor pressure)
- Scenario 5: Expanding to a second location

Running them in order simulates a real customer's journey over several months.

### Using the Optional Objections
Each scenario includes an **Optional Objection** in the Trainer Notes. Use these to add difficulty once an SC has the basic flow down. Don't introduce objections in the first run — let SCs build confidence before adding pressure.

## Trainer Tips

**Start every scenario the same way:** Ask the SC to explain what their goal is for this call before the roleplay begins. SCs who can articulate "I'm trying to understand their current setup before I show anything" are on the right track.

**Watch for these red flags:**
- Jumping to the demo before asking any discovery questions
- Using technical jargon (VoIP, SIP, PBX, softphone, hosted telephony)
- Demoing features in isolation without connecting them to a business problem
- Not asking "what would that change for you?" before the demo

**The best SC behavior to reinforce:**
- Asking one clear question, then listening fully before asking another
- Pausing after Taylor says something important and reflecting it back
- Letting Taylor "discover" the problem himself rather than stating it for him
- Ending every demo by asking: "Does this solve the problem you described?"

**Debrief questions after each scenario:**
1. At what point did you decide to move from discovery to demo?
2. What was the moment Taylor showed he was engaged?
3. What would you do differently if he had pushed back on the price?
4. Which feature mattered most to Taylor in this scenario — and how do you know?

---

## Assessment: Capstone Run

For final SC assessment, run all 5 scenarios back-to-back in a single 60–90 minute session. Taylor's situation should evolve naturally from each scenario to the next. Evaluate on:

- [ ] Discovery before demo in every scenario
- [ ] No jargon used without explanation
- [ ] Features connected to business outcomes
- [ ] Natural handling of at least one objection
- [ ] Confident use of GoTo Connect admin and demo environment
